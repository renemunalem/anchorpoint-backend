# MySQL Plan Freeze

Status: frozen for later restore. Active dev now defaults to PostgreSQL; JSON remains available only as a lightweight fallback/bootstrap path.

### What is incomplete / TODO

Not done yet:
- No dedicated MySQL Docker service is wired into the normal AtlasAI dev stack.
- No migration tool is installed. Current setup uses a raw schema SQL file and a seed/init script.
- No automated test suite runs against MySQL yet.
- No environment-specific health check or startup guard exists for `REPO_DRIVER=mysql` beyond config validation.
- No rollback/cleanup script exists beyond resetting the MySQL schema by rerunning the seed script.
- No commit has been made yet to isolate the MySQL work as a final backend-only checkpoint.

### Assumptions made

Persistence assumptions:
- PostgreSQL remains the default local dev persistence.
- JSON remains available with `REPO_DRIVER=json` only for lightweight fallback/bootstrap use.
- MySQL is opt-in only via `REPO_DRIVER=mysql`.
- Seed data source of truth remains the existing TS seed state under `src/data/*`.

Schema/tooling assumptions:
- Table names are plain snake_case tables, not generated by an ORM.
- No ORM is used.
- No migrations framework is used yet.
- Schema initialization is done by `db/mysql/schema.sql` plus `scripts/init-mysql.ts`.
- Timestamps are currently stored as ISO strings to match existing JSON behavior closely.

## B. How To Restore MySQL Later

### Required env vars

Exact names:
- `REPO_DRIVER=mysql`
- `MYSQL_HOST`
- `MYSQL_PORT`
- `MYSQL_DATABASE`
- `MYSQL_USER`
- `MYSQL_PASSWORD`
- `MYSQL_CONNECTION_LIMIT`

Example:

```bash
REPO_DRIVER=mysql
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_DATABASE=atlasai
MYSQL_USER=atlasai
MYSQL_PASSWORD=change_me
MYSQL_CONNECTION_LIMIT=10
```

### Docker Compose additions needed

Add a MySQL service to the AtlasAI dev compose stack similar to:

```yaml
atlasai-mysql:
  image: mysql:8.4
  environment:
    MYSQL_ROOT_PASSWORD: change_me
    MYSQL_DATABASE: atlasai
    MYSQL_USER: atlasai
    MYSQL_PASSWORD: change_me
  ports:
    - "3307:3306"
  volumes:
    - atlasai_mysql_data:/var/lib/mysql
```

Add backend env when switching to MySQL mode:

```yaml
atlasai-backend:
  environment:
    - REPO_DRIVER=mysql
    - MYSQL_HOST=atlasai-mysql
    - MYSQL_PORT=3306
    - MYSQL_DATABASE=atlasai
    - MYSQL_USER=atlasai
    - MYSQL_PASSWORD=change_me
    - MYSQL_CONNECTION_LIMIT=10
```

Add volume:

```yaml
volumes:
  atlasai_mysql_data:
```

### Migrations / schema initialization commands

Use the included schema and seed script:

```bash
cd /Users/rene/ai-dev-workspace/atlasai-backend
MYSQL_HOST=127.0.0.1 \
MYSQL_PORT=3307 \
MYSQL_DATABASE=atlasai \
MYSQL_USER=atlasai \
MYSQL_PASSWORD=change_me \
npm run db:mysql:init
```

This command:
- creates required tables if missing
- clears seeded rows
- reseeds users, members, cases, timeline, and RBAC permissions

### How to run backend in mysql mode

```bash
cd /Users/rene/ai-dev-workspace/atlasai-backend
REPO_DRIVER=mysql \
PORT=8082 \
MYSQL_HOST=127.0.0.1 \
MYSQL_PORT=3307 \
MYSQL_DATABASE=atlasai \
MYSQL_USER=atlasai \
MYSQL_PASSWORD=change_me \
npm run dev
```

### Data mapping from JSON seed state

Source of truth:
- `src/data/seedState.ts`
- `src/data/users.ts`
- `src/data/members.ts`
- `src/data/cases.ts`
- `src/data/rbac.ts`

Mapping rules:
- JSON `SeedUser.firstName` -> `users.first_name`
- JSON `SeedUser.lastName` -> `users.last_name`
- JSON `Member.subscriberMemberId` -> `members.subscriber_member_id`
- JSON `Member.phoneNumber` -> `members.phone_number`
- JSON `Member.addressLine1` -> `members.address_line1`
- JSON `Member.zipCode` -> `members.zip_code`
- JSON `Member.accountGroupName` -> `members.account_group_name`
- JSON `Member.cobCoverageTypes` -> JSON string in `members.cob_coverage_types`
- JSON `CaseDetail.caseNumber` -> `cases.case_number`
- JSON `CaseDetail.actionItem` -> `cases.action_item`
- JSON `CaseDetail.urgency.label` -> `cases.urgency_label`
- JSON `CaseDetail.urgency.tone` -> `cases.urgency_tone`
- JSON timeline `to` field -> `case_timeline.recipient_to`
- JSON RBAC `permissions` map -> JSON string in `rbac_permissions.permissions`

### Error handling and fallback behavior

Current behavior:
- No automatic fallback from MySQL to JSON if MySQL is selected and fails.
- If `REPO_DRIVER=mysql`, config is validated first.
- Missing required MySQL env values fail fast during startup.
- Route contracts and error shapes remain unchanged because the service/controller layer is repo-agnostic.
- JSON remains an explicit fallback when `REPO_DRIVER=json` is set.
